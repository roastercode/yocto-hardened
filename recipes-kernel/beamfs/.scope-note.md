# Scope note: yocto-hardened ↔ BEAMFS

**Date of decision:** 2026-04-29 evening session
**Decision author:** Aurelien Desbrieres
**Status:** binding for the current phase of the BEAMFS project

## Historical context

This layer (`yocto-hardened`) carried on-disk BEAMFS kernel sources under
`recipes-kernel/beamfs/files/beamfs-0.1.0/` during the late-April 2026
BEAMFS v2 development period. Those sources were lockstep-copied from
upstream `roastercode/beamfs` but **never tracked in this layer's git
history**. They were build inputs for an experimental dual-layer setup
(`yocto-hardened` + `yocto-beamfs` both in BBLAYERS) that mixed hardening
overlays with the BEAMFS research kernel module.

## Architectural decision

BEAMFS leaves `yocto-hardened`. The two layers are now disjoint in scope:

| Layer | Role | BEAMFS? |
|------|------|---------|
| `roastercode/yocto-beamfs` | BEAMFS research lab (qemuarm64), kernel module + RadFI fault-injection module | YES — sole BEAMFS Yocto layer |
| `roastercode/yocto-hardened` | FTRFS legacy + SELinux + dm-verity production-grade overlay | NO — production hardening only |

Hardening (SELinux + dm-verity + secure-boot + read-only rootfs) is
**explicitly out of scope** at the current stage of the BEAMFS project.
The current focus is the science:

  1. INLINE multi-block write path (sub-steps 2c–9)
  2. xfstests "auto" passing
  3. LOC audit < 5000 (DO-178C / IEC 61508 / MIL-STD-883)
  4. RadFI head-to-head bench (BEAMFS vs ext4 vs XFS under live EM
     perturbation)
  5. fs-devel RFC submission (anti-NAK Eric Biggers / Pedro Falcato)

Hardening overlays are a deployment concern that comes **after** the
filesystem itself is upstream-acceptable and scientifically defended.

## Re-introduction path

BEAMFS will return to `yocto-hardened` (or a successor production layer)
when the BEAMFS project reaches production maturity:

  - Full POSIX semantics (link, rename, truncate, fsync, mmap, xattr, ACL)
  - xfstests "auto" group passing under load
  - SELinux policy authored and validated for BEAMFS-specific labels
  - dm-verity-equivalent verified-boot path for BEAMFS
  - LOC audit per certification regimes finalized
  - fs-devel patch series accepted (post-NAK cycles)
  - First production deployment scenario greenlit

At that point, this scope note will be superseded by an explicit
hardening recipe in `recipes-kernel/beamfs-hardened/` referencing the
upstream BEAMFS sources via the standard recipe lockstep mechanism.

### Cleanup policy

These entries will be removed in a single explicit `chore(cleanup)`
commit if and when the decision is made to fully sterilize the layer
of BEAMFS residue. Until then they remain on disk as audit witnesses
of the dual-layer development period. They cannot be picked up by any
build because this layer is no longer in BBLAYERS.

Before any cleanup commit, verify each entry against
`roastercode/yocto-beamfs` and `roastercode/beamfs` to ensure no
unique content is lost. The only entry currently lacking a known
upstream lockstep is `Documentation/beamfs-integration.md` (this
layer's integration notes), which should be reviewed and either
ported upstream or archived to a BEAMFS-internal context note before
deletion.
